# Asset Organizer (`asset_organizer`)

A Frappe v15/v16 module inside `mpd_customizations` that tracks the full procurement lifecycle of fixed assets — from the first supplier quote through to commissioning — by attaching, classifying, and AI-extracting data from procurement documents (PDFs).

---

## Overview

When a capital asset is being purchased, dozens of documents are generated: quotes, purchase orders, tax invoices, gate passes, weighment slips, lorry receipts, e-Way bills, and payment advices. This module gives each asset a single **Asset Procurement Record (APR)** that collects all those documents, runs them through an LLM pipeline to extract structured data, and surfaces key milestones and financials automatically.

---

## DocTypes

### `Asset Procurement Record` (APR) — primary document

| Section | Fields |
|---|---|
| Asset Details | `asset_description` (required), `record_status` (auto-managed), `ai_description_suggestion` |
| Supplier | `supplier_doctype`, `supplier_link` (dynamic link), `supplier_gstin`, `supplier_name_raw` |
| Milestones | `quote_date`, `po_date`, `po_number`, `invoice_date`, `invoice_number`, `igp_date`, `igp_number` |
| Financials | `po_total_value`, `invoice_taxable_value`, `invoice_gst_amount`, `invoice_total_value`, `total_amount_paid`, `outstanding_balance` |
| Location | `current_location`, `final_location`, `installation_status`, `installation_date`, `erpnext_asset_link` |
| Child tables | `evidence_documents`, `item_lines`, `payments`, `location_log` |

**Naming:** `APR-.YYYY.-.#####`  
**Status flow (auto-managed):** Draft → Quotation Captured → PO Raised → Invoiced → Goods on Site → Payment Pending → Partially Paid → Fully Settled → Insurance Ready → Asset Commissioned

### `APR Evidence Document` — child table of APR

Represents one attached document (PDF).

| Field | Purpose |
|---|---|
| `evidence_file` | Frappe Attach field (local or S3) |
| `detected_category` | Quote / PO / Invoice / IGP / Weighment Slip / Lorry Receipt / E-Way Bill / Payment / Other |
| `document_description` | Plain-English label generated by the LLM |
| `extracted_date` | Document date extracted by LLM |
| `extracted_ref_no` | Reference number (PO number, invoice number, etc.) |
| `extraction_status` | Queued → Processing → Extracted / Failed |
| `ai_model_used` | Model identifier used for this extraction |
| `extracted_at` | Timestamp of extraction completion |
| `extraction_error` | Error message if extraction failed |
| `extracted_html_summary` | Human-readable HTML table of all extracted fields |
| `extracted_json_vault` | Full JSON blob of all extracted fields |
| `generic_data_map` | Key-value pairs for categories without a specific schema |

### `APR Item Line` — child table of APR

Line items consolidated from all extracted documents (invoices, POs).

| Field | Purpose |
|---|---|
| `raw_description` | Item description as it appears on the document |
| `hsn_code` | HSN/SAC code |
| `qty` | Quantity |
| `uom` | Unit of measure |
| `rate` | Unit rate |
| `source_document` | Which evidence document this line came from |

### `APR Payment` — child table of APR

Payment entries extracted from bank advice documents or added manually.

| Field | Purpose |
|---|---|
| `payment_date` | Date of payment |
| `amount` | Amount paid |
| `payment_mode` | RTGS / NEFT / Cheque / etc. |
| `reference_number` | UTR / cheque number |
| `remarks` | Free text |

### `APR Location Log` — child table of APR

Tracks where an asset has been as it moves from supplier to final installation.

| Field | Purpose |
|---|---|
| `log_date` | Date of location change |
| `location` | Location description |
| `logged_by` | User who recorded the entry |
| `remarks` | Free text |

### `Staged Supplier` — standalone DocType

A lightweight supplier record used when the actual supplier does not yet exist in ERPNext. The APR's `supplier_doctype` field can point to either `Supplier` (ERPNext) or `Staged Supplier`.

## AI Extraction Pipeline

### How it works

1. A user attaches a PDF to the `evidence_documents` table of an APR and saves.
2. The `Asset Procurement Record` controller (`on_update`) detects new rows with `extraction_status = "Queued"` and enqueues a background job per row onto the `long` queue.
3. The background job (`run_evidence_extraction` in `ai/apr_extraction.py`) runs two sequential LLM calls:
   - **Classification call** — identifies the document category and optionally suggests an asset description.
   - **Extraction call** — uses a category-specific schema and system prompt to extract all structured fields.
4. Results are written back to the child row via `frappe.db.set_value`, and the APR's header fields (dates, amounts, supplier info) are rolled up from all extracted evidence.

### File reading

Files are read before being sent to the LLM:
- **Local filesystem** (`/private/files/` or `/files/`) — read directly from disk.
- **S3 via `frappe_s3_attachment`** (`/api/method/frappe_s3_attachment.controller.generate_file?key=...`) — key is parsed from the URL and the file is downloaded directly using `S3Operations.read_file_from_s3()` (no HTTP round-trip to the Frappe web server, which does not resolve in background workers).

### LLM configuration

All LLM calls go through `mpd_base`'s `call_llm` utility, which reads configuration from `AI Task Config` DocType records. Each task key maps to one record:

| Task Key | Document Type |
|---|---|
| `apr_document_classify` | Classification (all categories) |
| `apr_extract_quote` | Supplier Quotation |
| `apr_extract_po` | Purchase Order |
| `apr_extract_invoice` | Tax Invoice |
| `apr_extract_igp` | Inward Gate Pass |
| `apr_extract_weighment` | Weighment Slip (RST) |
| `apr_extract_lorry_receipt` | Lorry Receipt (LR/GCN) |
| `apr_extract_eway_bill` | GST e-Way Bill |
| `apr_extract_generic` | Other / fallback |

The `AI Task Config` records are seeded by the patch `patches/seed_apr_ai_task_configs.py`.

### Pydantic schemas (`ai/schemas.py`)

Each document category has a typed Pydantic v2 schema:

| Schema class | Category |
|---|---|
| `ClassificationSchema` | Used for the first-pass classification call |
| `QuoteExtractionSchema` | Quote |
| `POExtractionSchema` | PO |
| `InvoiceExtractionSchema` | Invoice |
| `IGPExtractionSchema` | IGP |
| `WeighmentSlipSchema` | Weighment Slip |
| `LorryReceiptSchema` | Lorry Receipt |
| `EWayBillSchema` | E-Way Bill |
| `PaymentExtractionSchema` | Payment |
| `GenericExtractionSchema` | Other |

`CATEGORY_SCHEMA_MAP` and `CATEGORY_TASK_KEY_MAP` map category strings to the relevant class and task key respectively.

## API Endpoints (`api/apr.py`)

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `requeue_evidence_extraction(evidence_row_name)` | Reset a failed row to Queued and re-enqueue extraction. Requires System Manager or Stock Manager role. |
| `GET` | `get_apr_summary(apr_name)` | Returns a dashboard dict with status, financials, evidence counts by category, and a milestone checklist. |

Both are `@frappe.whitelist()` methods callable from the client script.

## Background Worker

The extraction jobs are enqueued to the `long` queue. The default Frappe `Procfile` worker only processes the `default` queue. A dedicated worker must be running:

```
worker_long: bench worker --queue long 1>> logs/worker.long.log 2>> logs/worker.long.error.log
```

This line is already present in the project `Procfile`.

## Setup / Installation

### 1. Run migrations

```bash
bench --site <site> migrate
```

This creates all DocType tables and runs registered patches.

### 2. Seed AI Task Config records

If the patch did not run automatically:

```bash
bench --site <site> execute mpd_customizations.asset_organizer.patches.seed_apr_ai_task_configs.execute
```

### 3. Ensure a long-queue worker is running

```bash
bench worker --queue long
```

Or start all processes via `make start-dev` (uses the updated `Procfile`).

### 4. Configure an LLM in AI Task Config

Open each `AI Task Config` record in the Frappe UI and set:
- **Model** — e.g. `openai/gpt-4o` or any OpenRouter model
- **API Key** — stored in the config or via environment variable as expected by `mpd_base`'s `call_llm`

## Document Category Reference

| Category | What it is | Key extracted fields |
|---|---|---|
| **Quote** | Supplier quotation or proforma | Supplier, date, total, item lines |
| **PO** | Purchase order from buyer | PO number, date, supplier GSTIN, item lines, total |
| **Invoice** | Tax invoice from supplier | Invoice number, date, taxable value, GST, total |
| **IGP** | Inward Gate Pass (goods receipt) | IGP number, date, vehicle number, receiver |
| **Weighment Slip** | Weigh bridge RST slip | RST number, gross/tare/net weight (kg), vehicle, material |
| **Lorry Receipt** | Transport LR / GCN | LR number, transporter, consignor/consignee GSTIN, from/to, freight |
| **E-Way Bill** | GST e-Way Bill | EWB number, IRN, validity, party GSTINs, taxable value, GST, item lines |
| **Payment** | Bank advice / RTGS confirmation | Date, amount, mode, UTR reference |
| **Other** | Anything unrecognised | Key-value generic extraction |
